## P0 — CRITICAL

### SEC-001: Central AI container mounts entire HOME directory rw
- **Files**: `src/process/container.ts:158-160`, `src/server/routes-admin.ts:50`
- **Impact**: Full read-write access to SSH keys, AWS/GCP/Azure creds, git tokens, all user data
- **Exploitability**: HIGH — any code running in Central AI container can exfiltrate all credentials
- **Recommendation**: Scope mounts to `~/.evomesh/central/` + project directories only. Use SSH agent forwarding instead of key file mounts.
- **Status**: OPEN

### SEC-002: SSH private keys mounted into ALL role containers
- **Files**: `src/process/container.ts:175-179`
- **Impact**: Every role container can read SSH private keys (ro mount of ~/.ssh)
- **Exploitability**: MEDIUM — requires container compromise, but any malicious role code gets keys
- **Recommendation**: Mount only `~/.ssh/known_hosts`. Use SSH agent forwarding (`SSH_AUTH_SOCK`) for git operations.
- **Status**: OPEN

## P1 — HIGH

### SEC-003: Admin API endpoints missing role authorization
- **Files**: `src/server/routes-admin.ts:108,120`
- **Endpoints**: `GET /api/admin/central-status`, `POST /api/admin/message`
- **Impact**: Any authenticated user (not just admin) can read Central AI status and send commands to it
- **Note**: Auth middleware in index.ts:105 ensures authentication, but these endpoints don't verify `session.role === "admin"` unlike `/api/admin/status` (line 86) and `/api/admin/start` (line 98) which do check.
- **Recommendation**: Add admin role check consistent with sibling endpoints.
- **Status**: OPEN

### SEC-004: Central AI uses host network mode
- **File**: `src/process/container.ts:152`
- **Impact**: Container shares host network namespace, can access all localhost services (databases, internal APIs, other containers)
- **Recommendation**: Use bridge network with explicit port mappings where possible.
- **Status**: OPEN

### SEC-005: All role containers mount project root rw
- **File**: `src/process/container.ts:164`
- **Impact**: Any role can modify the entire project source, including other roles' ROLE.md, todo.md, inbox — allows privilege escalation through file manipulation.
- **Note**: This is architecturally intended (roles collaborate via files), but a compromised role can tamper with any other role's instructions.
- **Status**: OPEN (accepted risk, but should be documented)

## P2 — MEDIUM

### SEC-006: Weak password policy
- **File**: `src/server/index.ts:69,92,173,198`
- **Impact**: Minimum 4 characters — trivially brute-forceable
- **Recommendation**: Minimum 8 characters, or integrate rate limiting.
- **Status**: OPEN

### SEC-007: No session expiry
- **File**: `src/server/index.ts:42`
- **Impact**: Session tokens in `Map<string, SessionInfo>` never expire. Stolen tokens remain valid indefinitely.
- **Recommendation**: Add TTL to sessions (e.g. 24h), rotate on use.
- **Status**: OPEN

### SEC-008: No rate limiting on login
- **File**: `src/server/index.ts:78-86`
- **Impact**: No throttling on `/auth/login` — allows unlimited brute-force attempts.
- **Recommendation**: Add per-IP rate limiting (e.g. 5 attempts/minute).
- **Status**: OPEN

### SEC-009: Token in localStorage
- **File**: `src/server/frontend.js:1-3`
- **Impact**: Auth token stored in localStorage is accessible to any XSS — if XSS exists, token is stolen.
- **Note**: Current XSS mitigations (esc() + addEventListener) look solid, but defense-in-depth says use httpOnly cookies.
- **Status**: OPEN
