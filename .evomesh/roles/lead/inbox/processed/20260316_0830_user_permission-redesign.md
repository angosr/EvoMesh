# 需求：权限系统重新设计

## 背景
当前的权限系统过于简单（只有 admin/viewer），无法支持多用户、多项目的真实使用场景。需要系统性重新设计。

## 用户的初步想法（需要你自行分析和完善）

### 角色分层
- **管理员 (admin)**: 系统级管理，可以管理所有 Linux 用户、查看所有用户的所有项目
- **Owner**: 拥有其 Linux 用户下的所有权限（新增/删除/管理项目，授权其他用户）
- **普通用户 (member)**: 只能访问被 owner 授权的项目

### 关键设计点
1. 每个用户对应一个 Linux user（容器挂载时用对应用户的文件权限）
2. Owner 可以授权 member 访问自己的某个项目
3. "忽略项目"不等于删除——别人可能在看这个项目，只是从自己的工作区隐藏
4. 需要一个工作区管理界面：添加项目、隐藏项目、授权管理

## 要求

### 请你做以下工作：
1. **系统性分析**：分析当前权限系统的不足，对比主流工具的权限设计（VS Code Server、Jupyter Hub、Gitea 等）
2. **自我攻击**：对你设计的方案做攻击测试（按 ROLE.md 1.2 自审查协议），找出安全漏洞和 UX 问题
3. **设计完整交互流程**：
   - 管理员视角：怎么管理用户和系统设置
   - Owner 视角：怎么管理项目和授权
   - Member 视角：怎么加入/查看项目
   - 登录/注册流程
   - 项目权限矩阵
4. **输出设计文档**：写入 `docs/design-permission-system.md`（中英双语）
5. **分派给 executor 实施**：攻击通过后，把实施任务写入 executor 的 todo.md

### 设计原则
- 好用 > 功能多（宁可少几个功能，也不要把界面搞复杂）
- 安全 > 便利（权限默认最小化）
- 渐进式：可以分阶段实施，先做最核心的部分

### 重要提醒
- 你需要不停的反思当前的交互是否合理友善
- 当前的 Web UI 交互流程也需要一起审视（侧栏、项目树、设置页面等）
- 设计要考虑移动端适配
