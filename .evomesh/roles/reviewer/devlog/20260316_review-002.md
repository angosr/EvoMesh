## New Findings

### P1 ‚Äî Bug / Logic Error

#### P1-4: Terminal proxy has no authentication
**File**: `src/server/terminal.ts:70-81` (HTTP) and `84-114` (WebSocket)
**Issue**: The auth middleware in `index.ts:107` explicitly skips terminal paths:
```typescript
if (req.path.startsWith("/terminal/")) return next();
```
The terminal proxy itself does zero auth checking. Anyone who knows/guesses the URL pattern `/terminal/{slug}/{role}/` can access any running terminal ‚Äî full shell access.
The WebSocket handler (line 83) has a comment "no auth needed ‚Äî ttyd binds localhost only" but this is wrong: the proxy exposes it to the network.
**Risk**: CRITICAL. An unauthenticated user can access any running role's terminal.
**Fix**: Extract and validate the Bearer token (or token query param) in both the HTTP and WebSocket proxy paths before forwarding.

#### P1-5: spawner.ts writes shell script with unsanitized role name
**File**: `src/process/spawner.ts:192-222`
**Issue**: The send script uses env vars (good ‚Äî `EVOMESH_SESSION`, `EVOMESH_LOOP_CMD`), so the script body itself is safe from injection. However, the `tmuxSessionName` function on line 18 uses `projectSlug` which comes from `slugify(path.basename(root))` ‚Äî if `slugify` doesn't properly sanitize, this could be a problem. Self-attack: `slugify` appears to replace non-alphanumeric with hyphens, so this is likely safe. **Low risk.**

#### P1-6: showRoleModal and doCreateRole still use inline onclick
**File**: `src/server/frontend.js:401-403`
**Issue**: Lines 401-403 still have inline event patterns:
```js
function showRoleModal(slug) { ... document.getElementById('role-modal-overlay').classList.add('show'); ... }
```
The `showRoleModal` is called from an `addEventListener` via the sidebar "Add Role" button, so the call site is safe. But `closeRoleModal` is still called from an `onclick` attribute in the HTML template (`frontend.html`). This is a remnant ‚Äî lower risk since the slug doesn't flow into it, but inconsistent.
**Self-attack**: The HTML onclick for closeRoleModal has no dynamic data, so XSS risk is nil. **Cosmetic.**

### P2 ‚Äî Quality Improvement

#### P2-5: Dockerfile uses ubuntu:24.04 without pinning digest
**File**: `docker/Dockerfile:1`
**Issue**: `FROM ubuntu:24.04` uses a floating tag. A compromised or updated base image could introduce vulnerabilities. Also:
- `npm install -g @anthropic-ai/claude-code@latest` uses `@latest` ‚Äî non-reproducible builds
- `pip3 install --break-system-packages websockets` ‚Äî no version pin
- `nodejs` from Ubuntu repos is typically outdated
**Fix**: Pin image digest, pin package versions for reproducibility.

#### P2-6: Dockerfile installs unnecessary `python3-pip` + `websockets`
**File**: `docker/Dockerfile:4,10`
**Issue**: Python and websockets are installed but don't appear to be used anywhere in entrypoint.sh or by Claude Code. Appears to be leftover from a previous approach.
**Self-attack**: Maybe used by Claude Code internally or by the central AI? Need to confirm.

#### P2-7: terminal.ts empty catch blocks suppress all errors
**File**: `src/server/terminal.ts:37,48,50`
**Issue**: Three nested `try/catch {}` blocks with empty catches. If container state detection fails, the error is silently swallowed. At minimum, add debug logging.

#### P2-8: spawner.ts send script never cleans up
**File**: `src/process/spawner.ts:224-235`
**Issue**: The detached background script (`${roleName}-send.sh`) is `unref()`'d. If the parent Node process exits before the script completes, the script continues running as orphan. The script file itself is never deleted. Minor ‚Äî files are small and in a runtime dir.
