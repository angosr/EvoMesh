## Findings

### P0 — Security / Crash

#### P0-1: XSS via innerHTML in frontend.js (CRITICAL)
**File**: `src/server/frontend.js` multiple locations
**Issue**: The `esc()` function is used for HTML escaping, but several places construct HTML with string interpolation that bypasses it. For example:
- Line 160: `onclick="...closePanel('${key.replace(/'/g, "\\'")}')"` — if a key contains `">`, this breaks out of the attribute.
- Line 191-193: Account select `onchange` and action buttons build `onclick` handlers with `esc()` which doesn't escape single quotes in JS context.
- Line 424: `onmousedown="event.preventDefault();selectCompletion('${sp}')"` — `sp` only escapes `'` but not `"` or other HTML-attribute-breaking chars.
**Risk**: An attacker who can control role names or project slugs could inject arbitrary JS.
**Fix**: Use `addEventListener` instead of inline `onclick` handlers, or properly escape for both HTML attribute and JS string context.

#### P0-2: Central AI mounts entire HOME directory read-write
**File**: `src/server/routes-admin.ts:49`
```typescript
"-v", `${homeDir}:${homeDir}:rw`,
```
**Issue**: The Central AI container has full read-write access to the user's entire home directory, including SSH keys, credentials, and other sensitive files.
**Fix**: Mount only the necessary directories (similar to normal role containers in `container.ts:163-167`).

#### P0-3: SSH keys mounted into role containers
**File**: `src/process/container.ts:178-179`
```typescript
const sshDir = path.join(homeDir, ".ssh");
if (fs.existsSync(sshDir)) { args.push("-v", `${sshDir}:${sshDir}:ro`); }
```
**Issue**: While read-only, mounting SSH keys into containers exposes private keys to any code running inside. A compromised container could exfiltrate keys.
**Self-attack**: Needed for git operations with SSH remotes. Consider mounting only public keys or using SSH agent forwarding instead.

#### P0-4: EventSource auth bypass for SSE refresh endpoint
**File**: `src/server/routes.ts:95-101`, `src/server/frontend.js:648`
**Issue**: The `/api/refresh/subscribe` SSE endpoint doesn't validate the token query param server-side. The frontend passes it: `new EventSource(\`\${API}/refresh/subscribe?token=\${encodeURIComponent(AUTH_TOKEN)}\`)` but the server endpoint (`_req`) ignores it. Any unauthenticated client can subscribe to refresh events.
**Risk**: Low data leakage (only refresh pings), but pattern could be replicated to more sensitive SSE endpoints.

### P1 — Bug / Logic Error

#### P1-1: `sendInput` uses shell injection-vulnerable string interpolation
**File**: `src/process/container.ts:267`
```typescript
execFileSync("docker", ["exec", name, "sh", "-c", `printf '%s\n' "${input.replace(/"/g, '\\"')}" > /proc/1/fd/0`], ...)
```
**Issue**: The escaping only handles double quotes. Input containing `$()`, backticks, or `\` could result in command injection inside the container's shell.
**Fix**: Use a safer approach — write to a temp file or use `docker exec` with explicit args instead of `sh -c`.

#### P1-2: Race condition in port allocation
**File**: `src/server/routes.ts:72-76`
```typescript
export function allocatePort(ctx: ServerContext): number {
  let port = ctx.port + 1;
  for (const [, t] of ctx.ttydProcesses) { if (t.port >= port) port = t.port + 1; }
  return port;
}
```
**Issue**: Two concurrent start requests could allocate the same port since `ttydProcesses.set()` happens after `allocatePort()` returns. No locking mechanism.
**Fix**: Use an atomic counter or immediately reserve the port in the map.

#### P1-3: Unused imports in routes.ts
**File**: `src/server/routes.ts:5,10-16,18`
**Issue**: After the refactor splitting routes into 3 files, `routes.ts` still imports many symbols that are only used in `routes-roles.ts` and `routes-admin.ts`:
- `createRole`, `deleteRole` (line 10) — used in `routes-roles.ts`
- `TEMPLATES`, `TEMPLATE_NAMES` (line 11) — used in `routes-roles.ts`
- `startRole`, `stopRole`, `restartRole`, `isRoleRunning`, `getRoleLogs`, `sendInput`, `switchAccount`, `getContainerState`, `getContainerPort` — partially used
- `ensureTtydRunning` (line 18) — only used in `index.ts`
**Fix**: Remove unused imports.

### P2 — Quality Improvement

#### P2-1: Pervasive `e: any` catch pattern
**Files**: All route files (27+ occurrences)
**Issue**: Every catch block uses `(e: any)` instead of proper error typing. This defeats TypeScript's type system.
**Fix**: Use `(e: unknown)` and a helper like `function errorMessage(e: unknown): string { return e instanceof Error ? e.message : String(e); }`.

#### P2-2: `loadConfig()` called repeatedly per request
**Files**: `routes.ts`, `routes-roles.ts`
**Issue**: Multiple endpoints call `loadConfig()` multiple times — e.g., `routes-roles.ts:141` calls `loadConfig` twice in the config endpoint (once to read, once fresh after stop). While the YAML is small, this is inefficient and could cause TOCTOU issues.
**Self-attack**: Files are small, perf impact minimal. Low priority.

#### P2-3: Frontend JS has no build/minification
**File**: `src/server/frontend.js` (655 lines, raw JS)
**Issue**: The frontend is served as raw unminified JS. No TypeScript, no bundling, no tree-shaking.
**Self-attack**: For an internal tool this is fine. Would only matter at scale.

#### P2-4: `acl.yaml` read on every auth check
**File**: `src/server/acl.ts:61` — `loadAcl()` reads file from disk on every `getProjectRole()` call.
**Issue**: Every authenticated API request triggers a disk read of `acl.yaml`. Should cache in memory and invalidate on write.
**Self-attack**: File is tiny, only one user likely. Acceptable for now.
